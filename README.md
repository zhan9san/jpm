# jpm — Jenkins Plugin Manager

Generates reproducible Jenkins plugin lock files from a `plugins.txt` manifest.

Resolves all transitive plugin dependencies against the Jenkins Update Center and
writes a `plugins-lock.txt` with every plugin pinned to an exact version — like
`Cargo.lock` or `package-lock.json`, but for Jenkins.

## Requirements

- [Rust](https://rustup.rs/) 1.75+

## Build

```bash
# Debug build (fast compile, slower binary)
cargo build

# Release build (optimised binary)
cargo build --release
```

The binary is written to `target/debug/jpm` or `target/release/jpm`.

You can also install it directly to `~/.cargo/bin/` so it is on your `PATH`:

```bash
cargo install --path .
```

## Run

```bash
jpm --jenkins-version <VERSION> [OPTIONS]
```

### Options

| Flag | Short | Default | Description |
|------|-------|---------|-------------|
| `--jenkins-version` | `-j` | *(required)* | Jenkins version to target (e.g. `2.452.4`) |
| `--plugin-file` | `-f` | `plugins.txt` | Path to the input plugin manifest |
| `--output` | `-o` | `plugins-lock.txt` | Path for the generated lock file |
| `--skip-bundled` | | `false` | Skip fetching bundled plugins from WAR pom.xml |

### Examples

```bash
# Resolve plugins.txt against Jenkins 2.452.4, write plugins-lock.txt
jpm --jenkins-version 2.452.4

# Explicit file paths
jpm -j 2.452.4 -f my-plugins.txt -o my-lock.txt

# Skip bundled plugin detection (faster, slightly less accurate)
jpm -j 2.452.4 --skip-bundled
```

Using `cargo run` directly (no separate build step needed):

```bash
cargo run -- --jenkins-version 2.452.4
cargo run -- -j 2.452.4 -f plugins.txt -o plugins-lock.txt
```

## Input format — `plugins.txt`

One plugin per line. Lines starting with `#` are comments; inline `#` comments
are also supported.

```text
# format: name[:version[:url]]
git
credentials:1453.v9b_a_29777a_b_fd
workflow-aggregator:latest
junit:experimental
script-security::http://my-mirror.example.com/script-security.hpi
```

| Version value | Behaviour |
|---------------|-----------|
| omitted or `latest` | Resolve to the current latest version |
| `experimental` | Resolve from the experimental update center |
| anything else | Use that exact version |

## Output format — `plugins-lock.txt`

```text
# Generated by jpm — do not edit manually
# Jenkins: 2.452.4

apache-httpcomponents-client-4-api:4.5.14-208.v438351942757
credentials:1453.v9b_a_29777a_b_fd
git:5.7.0
git-client:6.1.0
...
```

- Sorted alphabetically for clean diffs
- Includes all transitive dependencies
- Plugins already bundled in the Jenkins WAR are listed with their bundled
  version (no redundant download)

## Caching

Update Center JSON responses are cached in `~/.cache/jpm/` with a **1-hour TTL**.
Delete that directory to force a fresh fetch.

## How it works

1. Parse `plugins.txt` → list of plugin requests
2. Fetch three Jenkins Update Center endpoints (cached):
   - `update-center.actual.json?version=<V>` — latest versions
   - `current/plugin-versions.json` — all pinned versions + deps
   - `experimental/update-center.actual.json` — experimental channel
3. Fetch bundled plugin versions from `jenkins-<VERSION>/war/pom.xml` on GitHub
4. BFS dependency resolution: higher version wins on conflicts, optional deps skipped
5. Write sorted `plugins-lock.txt`

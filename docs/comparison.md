# Package Manager Comparison: jpm vs npm vs pip-tools vs poetry

## Overview

All four tools implement the same two-file pattern: a human-edited **manifest**
that expresses intent, and a machine-generated **lock file** that pins every
dependency to an exact version for reproducible installs.

```text
Manifest (human-edited)       Lock file (machine-generated)
в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ      в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
package.json           +      package-lock.json       (npm)
requirements.in        +      requirements.txt    (pip-tools)
pyproject.toml         +      poetry.lock            (poetry)
plugins.txt            +      plugins-lock.txt          (jpm)
```

## File Formats

### Manifest

**npm вЂ” `package.json`**

```json
{
  "dependencies": {
    "lodash": "^4.17.0",
    "express": "~4.18.0"
  }
}
```

**pip-tools вЂ” `requirements.in`**

```text
requests>=2.28

```

**poetry вЂ” `pyproject.toml`**

```toml
[tool.poetry.dependencies]
requests = "^2.28"
flask = "~2.3"
```

**jpm вЂ” `plugins.txt`**

```text

credentials:1453.v9b_a_29777a_b_fd

```

## Dependency Resolution

| Aspect | npm | pip-tools | poetry | jpm |
|---|---|---|---|---|
| **Algorithm** | BFS + semver ranges | Backtracking | SAT solver (PubGrub) | BFS |
| **Version constraint** | semver (`^`, `~`, `*`) | PEP 440 | semver | Exact minimum from UC |
| **Conflict rule** | Highest in range | Highest compatible | Satisfies all | Highest version wins |
| **Optional deps** | Included by default | N/A | `extras` opt-in | Skipped |
| **Duplicate detection** | Yes | Yes | Yes | Yes вЂ” BFS skip |

### Why jpm uses "highest version wins"

Jenkins plugins declare dependencies as **exact minimum versions**, never as ranges:

```text
Plugin-Dependencies: git:5.2.0,credentials:1371.0
```

This means: "I need at least `git 5.2.0`." When two plugins both require `git`
at different minimum versions, the safe choice is always the higher one вЂ” it
satisfies both constraints. A SAT solver would reach the same conclusion with
zero additional benefit, so BFS with highest-wins is both correct and simpler.

## Integrity Verification

| Tool | Hash algorithm | Source | Verified on |
|---|---|---|---|
| npm | SHA-512 | npm registry | Install |
| pip-tools | SHA-256 | PyPI | `pip-sync` |
| poetry | SHA-256 | PyPI | `poetry install` |
| jpm | SHA-256 | Jenkins Update Center | `jpm install` |

jpm reads the `sha256` field directly from the UC's `plugin-versions.json` вЂ”
no local computation or extra HTTP requests needed. The hash is written into
`plugins-lock.txt` and verified at install time when each `.hpi` file is
downloaded.

## Staleness Detection

**poetry** stores a `content-hash` of `pyproject.toml` inside `poetry.lock`:

```toml
[metadata]
content-hash = "a8f1b2c4..."
```

**jpm** follows the same pattern with a `manifest-hash` in the lock file header:

```text
# manifest-hash: a3f1b2c4d5e6f7a8
```

Both tools warn when the manifest has changed since the lock was last generated:

```text
warning: 'plugins-lock.txt' is out of date вЂ” manifest changed since last lock
```

npm does not embed a manifest hash. It detects staleness indirectly by
comparing `package.json` modification times against `package-lock.json`.

## Download Strategy

| Tool | Concurrency | Download target |
|---|---|---|
| npm | Sequential by default | `node_modules/` |
| pip-tools | N/A (lock only; `pip-sync` installs) | site-packages |
| poetry | Sequential | virtualenv |
| Java plugin manager | 64-thread `ForkJoinPool` | `$JENKINS_HOME/plugins/` |
| jpm | Concurrent `tokio` tasks | `--plugin-dir` (default `./plugins/`) |

## Jenkins-Specific: Bundled Plugin Detection

This feature is unique to jpm. Modern Jenkins ships ~40 plugins bundled inside
`jenkins.war`. Installing an older version of a bundled plugin wastes a download
and can cause conflicts at startup.

Other package managers have no equivalent concept.

**How jpm handles it:** Fetches `war/pom.xml` from GitHub for the target Jenkins
version (single lightweight HTTP request вЂ” no 80 MB WAR download), parses
`<artifactItem>` blocks with `<type>hpi</type>`, and uses the bundled version
when it is equal to or newer than the requested version.

```text
https://raw.githubusercontent.com/jenkinsci/jenkins/jenkins-{VERSION}/war/pom.xml
```

The pom.xml for a tagged release is immutable, so it is cached permanently at
`~/.cache/jpm/pom-{version}.xml` with no TTL.

## Known Gaps vs npm / poetry

| Gap | Notes |
|---|---|
| Version ranges | Jenkins UC only exposes exact minimum constraints вЂ” ecosystem limitation |
| `jpm update <plugin>` | Targeted single-plugin update not yet implemented |
| Offline mode | No fallback when UC endpoints are unreachable |
| Direct vs transitive in lock | Tracked internally but not written to lock file |

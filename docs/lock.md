# jpm lock

`jpm lock` resolves `plugins.txt` into a reproducible `plugins-lock.txt`.

It is the "resolve and pin" step in JPM:

1. Read requested plugins from `plugins.txt`.
2. Fetch Update Center metadata (stable, experimental, plugin-versions).
3. Resolve transitive dependencies.
4. Check Jenkins-core compatibility.
5. Write exact versions and checksums into `plugins-lock.txt`.

## Command

```text
jpm lock -j <VERSION> [OPTIONS]

Options:
  -j, --jenkins-version <VERSION>  Target Jenkins version (required)
  -f, --plugin-file <FILE>         Input manifest [default: plugins.txt]
  -o, --output <FILE>              Output lock file [default: plugins-lock.txt]
      --skip-bundled               Skip WAR bundled plugin lookup
      --fix                        Rewrite only incompatible pinned versions
      --upgrade                    Rewrite all pinned versions to highest compatible
```

## Input format (`plugins.txt`)

Supported line formats:

```text
artifact
artifact:version
artifact:version:url
```

Version rules:

- Omitted version or `latest` -> resolve from stable Update Center.
- `experimental` -> resolve from experimental Update Center.
- Any other value -> treated as pinned exact version.

Comments and blank lines are allowed.

## What gets written (`plugins-lock.txt`)

The lock file contains:

- Header with target Jenkins version.
- `manifest-hash` of `plugins.txt` (staleness check).
- One line per resolved plugin: `name:version sha256:<base64>`.

Example:

```text
# Generated by jpm - do not edit manually
# Jenkins: 2.452.4
# manifest-hash: a3f1b2c4d5e6f7a8

credentials:1492.vcfb_5dc894a_e3 sha256:sGHJ272ebC0wsq0ktJVQnOeyWrKRE09cJ85EDi+3ewo=
git:5.10.0 sha256:RJexe7ZIFg6uk5Dyxikzx6urM82UDjAd7wcx0bLzWB4=
```

Plugins are sorted alphabetically for stable diffs.

## Compatibility behavior

During lock, JPM checks each resolved plugin version against
`--jenkins-version` using `requiredCore`.

- If incompatible plugins exist and neither `--fix` nor `--upgrade` is set:
  - lock fails with clear errors.
- `--fix`:
  - rewrites only incompatible pinned entries in `plugins.txt` to the highest
    compatible version.
- `--upgrade`:
  - includes `--fix` behavior and also upgrades all other pinned plugins to the
    highest compatible version.

After rewrite, JPM resolves again and writes the new lock.

## Bundled plugin handling

By default, JPM fetches bundled plugin versions for your Jenkins core and avoids
pinning older versions than what Jenkins already bundles.

Use `--skip-bundled` for faster runs if you do not need bundled-awareness.

## Staleness warning

When writing a lock, JPM compares the existing lock's `manifest-hash` with the
current `plugins.txt` hash.

If they differ, JPM warns that the lock is out of date.

## Dependency cycle design

`jpm lock` performs cycle detection on an effective plugin graph and fails fast
when a cycle is found.

### Graph inputs

- Resolved plugins from `plugins.txt` (including transitive dependencies).
- Bundled plugins for the target Jenkins core.
- Jenkins split/detached metadata:
  - `split-plugins.txt`
  - `split-plugin-cycles.txt`

### Edge rules

- Base dependency edges come from Update Center dependency metadata.
- Optional edges are considered active when the target plugin is present.
- Split/detached implied edges are added based on Jenkins core split rules.
- Cycle-break edges from `split-plugin-cycles.txt` are removed.

### Detection

- Build directed adjacency map from effective plugins and active edges.
- Run DFS cycle detection (back-edge check on visiting nodes).
- On cycle, abort lock and print a concrete path:

```text
found cycle in plugin dependencies: a -> b -> c -> a
```

### Known limitations

- Dependency metadata still comes from Update Center, not runtime plugin
  manifests.
- In edge cases, Jenkins runtime behavior can still differ when manifest data
  and Update Center data diverge.

## Typical usage

```bash
# Initial lock
jpm lock -j 2.541.3

# Auto-fix only incompatible entries
jpm lock -j 2.541.3 --fix

# Refresh everything to newest compatible versions
jpm lock -j 2.541.3 --upgrade
```
